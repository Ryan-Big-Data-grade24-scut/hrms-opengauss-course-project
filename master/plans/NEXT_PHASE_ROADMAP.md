# Next Phase Roadmap

这份路线图回答一个问题：

当前数据库、后端、前端最小闭环已经跑起来后，下一步最值得投入时间的是什么。

## 1. 当前基线回顾

我们已经具备：

- openGauss 数据库与最小 HRMS schema
- 可跑的 Python 后端
- 可跑的 Vue 前端
- 基本登录、员工、部门、岗位、请假、个人中心页面
- 后端 smoke test 和启动脚本

这意味着项目已经从“想法阶段”进入“功能完善阶段”。

## 2. 下一步优先级建议

### P0：先把现有系统做完整

这是最重要的，不要急着先上花活。

#### 前端

- 员工页补完整筛选体验
- 员工编辑弹窗字段校验
- 请假页补审批按钮
- 增加审计日志页面
- 增加用户/角色管理页

#### 后端

- 补用户管理接口的前端联调支持
- 补请假审批按钮对应接口联调
- 为前端补更稳定的错误码和错误提示
- 统一更多接口返回格式细节

#### 数据库

- 梳理字段是否需要补充注释和默认值
- 给 `employee` 增加更完整的人事字段候选
- 给 `leave_request` 增加审批人、审批时间候选字段
- 评估是否增加 `attendance`、`salary_stub`、`job_transfer` 等扩展表

## 3. 数据库字段升级建议

当前表结构够最小演示，但如果要做得更像“完整 HR 系统”，下一步建议优先补这些：

### `employee`

- `id_card_no`
- `birth_date`
- `address`
- `emergency_contact`
- `education_level`
- `employment_type`

### `department`

- `department_code`
- `description`

### `position`

- `position_code`
- `status`

### `leave_request`

- `approver_username`
- `approved_at`
- `approval_comment`

这些升级的价值是：

- 更像真实人力系统
- 更利于写报告和 ER 说明
- 后续更容易扩展 AI 查询和统计页

## 4. 前端下一阶段建议

推荐顺序：

1. 员工页做完整 CRUD 体验
2. 补审计页
3. 补用户/角色页
4. 补请假审批动作
5. 再做更亮眼的首页 dashboard

为什么这样排：

- 这几页最贴合数据库课设评分点
- 最容易形成“老师一眼看懂”的演示路径

## 5. 后端下一阶段建议

推荐顺序：

1. 补审计查询接口筛选
2. 补用户/角色前端所需接口
3. 对关键写操作做更严格参数校验
4. 预留备份恢复接口真实实现

如果还有时间，可以把数据库访问从“直接调 gsql”逐步升级到更正规的数据库驱动方案。但这不是当前最急的。

## 6. 拓展方向

### 6.1 备份恢复模块

这条线来自你们一开始就想做的亮点。

最务实的做法不是把整个项目变成“纯备份恢复系统”，而是：

- 保留 HRMS 为主系统
- 把备份恢复做成管理员专用模块

可做内容：

- 查看备份记录
- 手动触发逻辑备份
- 恢复入口占位
- 备份日志展示

### 6.2 AI 查询助手

这是最适合做展示亮点的 AI 方向。

建议做法：

- 只做只读查询
- 输入自然语言
- 后端转换成受控 SQL
- 展示最终 SQL 和结果

为什么适合：

- 亮点强
- 和数据库课程很贴
- 风险比“让 AI 直接接管数据库”低很多

### 6.3 AI 审计解释助手

比 AI 查库更稳的一条路。

可做内容：

- “过去 7 天谁删过员工？”
- “最近最频繁的操作是什么？”
- “这条审计记录是什么意思？”

这条线非常适合和 `audit_log` 结合。

## 7. 我建议的现实推进顺序

如果以“课程交付成功率最高”为目标，我建议：

1. 员工页补完
2. 请假审批页补动作
3. 审计页补出来
4. 用户/角色页补出来
5. 再决定做备份恢复还是 AI 查询亮点

## 8. 最后一句判断标准

下一阶段的工作应该始终围绕这句话：

> 让老师在 3 分钟内看明白，这是一套完整的 openGauss 数据库应用系统。

如果某个功能不能明显提升这件事，就不该排在最前面。
