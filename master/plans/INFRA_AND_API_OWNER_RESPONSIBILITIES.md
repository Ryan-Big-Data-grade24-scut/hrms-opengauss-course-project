# Infrastructure And API Owner Responsibilities

这份文档回答一个问题：

> 如果你负责“数据库基础设施 + 数据库架构设计 + 各模块接口设计 + 正式功能开发前的公共基建”，你到底要承担什么职责？

## 1. 角色定位

你不是单纯“写几个接口的人”。

你更接近：

- 数据库基础设施负责人
- 数据库架构与接口契约负责人
- 模块开发前公共基建负责人

也就是说，你的职责是：

- 把底层运行面搭稳
- 把模块边界定义清楚
- 把接口契约冻结出来
- 让后续每个功能模块都能在这个底座上并行开发

## 2. 你的核心职责

## 2.1 数据库基础设施

你负责保证：

- openGauss 能稳定启动
- 数据库初始化可重复执行
- 数据库验证可重复执行
- 数据不会因为单人临时环境而不可复现

你要交付：

- `ops/db/`
- `ops/startup/`
- `sql/` 基线脚本
- 数据库启动与验证手册

## 2.2 数据库架构设计

你负责定义：

- 核心实体
- 表关系
- 主键、外键、索引
- 基础字段规范
- 后续字段升级路线

你要交付：

- `master/DATABASE_SPEC.md`
- 后续迁移规划
- 模块字段字典

## 2.3 接口契约设计

你负责定义：

- 每个模块有哪些 API
- 输入输出结构
- 分页和筛选规则
- 错误码规范
- 权限要求

你要交付：

- `master/API_SPEC.md`
- 后续 `openapi.yaml`

## 2.4 开发前公共基建

在正式功能开发前，你应该先把这些东西准备好：

- 统一返回格式
- 统一鉴权逻辑
- 统一审计写入逻辑
- 统一启动脚本
- 统一 smoke test
- 统一目录职责

这样后面的模块开发才不会互相打架。

## 3. 你不应该无限承担的部分

你可以搭底座，但不应该永久兼任所有模块开发者。

建议你把边界控制在：

- 你负责“公共层”和“模块边界”
- 组员负责“各模块业务实现和页面联调”

否则风险是：

- 你一个人把所有决定都揽走
- 组员无法稳定接入
- 一旦你停下来，整个项目就断了

## 4. 什么叫 AI 友好 + 分工友好

## 4.1 AI 友好

一个 AI 友好的基础架构，至少要满足：

- 目录职责清晰
- 脚本可 CLI 启动
- 计划和文档位置固定
- API 和数据库规格可被单独读取
- 模块边界可被快速理解

## 4.2 分工友好

一个分工友好的基础架构，至少要满足：

- 数据库同学知道该改哪
- 后端同学知道接口契约是什么
- 前端同学知道字段和接口怎么用
- 文档同学知道哪些材料是正式提交物

## 5. 你这阶段最重要的交付物

按优先级排序：

1. 稳定启动链路
2. 冻结数据库基线
3. 冻结接口契约
4. 给每个模块定义边界
5. 给组员留下可接手的规范和脚本

## 6. 你接下来最该做的事

如果你继续承担这个角色，建议顺序是：

1. 把 `master/` 计划区整理清楚
2. 把 `API_SPEC.md` 升级成正式 OpenAPI
3. 把数据库脚本从“大 schema”逐步升级成迁移式
4. 给每个模块列出“开发前置条件”
5. 再把具体业务模块交给对应同学并行推进
