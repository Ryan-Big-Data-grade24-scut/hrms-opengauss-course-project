# Step 01：课程要求基线

## 这一步的目标

把“大家觉得老师要什么”变成“老师原文真正要什么”。

## 这一步完成后，必须得到的结论

1. 哪些是硬要求
2. 哪些只是推荐或配套资料
3. 这次项目最稳的主线是什么
4. 这次项目最小需要交付哪些东西

## 目前已经确认的结论

### 硬要求

- 题目可以自定义
- 3 到 4 人组队完成
- 每个成员要负责具体功能，并在文档中说明贡献
- 项目必须基于 `openGauss`
- 要提交：
  - 系统源代码和可执行文件（如适用）
  - 英文版数据库应用系统开发手册
  - 数据库文件
  - 不超过 3 分钟的演示视频

### 尚未发现的硬要求

- 未看到“必须使用华为云账号”
- 未看到“必须部署到华为云 ECS”
- 未看到“必须在线云开发”

### 由此得到的执行结论

- 数据库平台必须用 `openGauss`
- 部署环境优先走本地 Docker，不强依赖华为云
- 主线优先做标准数据库应用系统，不优先做复杂创新系统

## 这一步接下来要做什么

### 1. 冻结项目主线

目标：

- 定死你们到底做什么系统

建议：

- 先定成 `企业人力资源管理系统（openGauss）`

完成标准：

- 能用一句话说明项目名称、角色、核心功能

### 2. 冻结最小范围

目标：

- 防止项目越做越大

建议最小模块：

- 登录
- 用户/角色/权限
- 部门管理
- 岗位管理
- 员工管理
- 请假管理
- 审计日志

完成标准：

- 明确哪些模块第一阶段必须做
- 明确哪些模块第二阶段再补

### 3. 冻结技术边界

目标：

- 防止技术路线频繁变动

建议：

- 数据库：`openGauss`
- 环境：`Docker`
- 后端：`C++17 + CMake`
- 数据访问：`libpqxx`
- 前端：先以简单页面跑通为主

完成标准：

- 不再在 Java / Python / 小程序 / 云端部署之间反复摇摆

### 4. 准备 Step 02

Step 02 的目标是：

- 跑通 openGauss
- 建出最小数据库
- 插入演示数据

## 这一步谁该做什么

### 组长 / 主推进人

- 统一最终题目
- 冻结范围
- 确认每人负责模块

### 数据库负责人

- 准备 ER 图
- 准备建表清单

### 后端负责人

- 准备接口分层方案

### 前端负责人

- 准备最小页面清单

## 一句话判断是否完成

当你们能用 1 分钟清楚说出“老师要什么、我们做什么、第一阶段不做什么”时，这一步才算真的完成。
