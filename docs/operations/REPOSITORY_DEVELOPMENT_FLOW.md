# Repository Development Flow

这份文档讲的是：

不是系统“怎么运行”，而是我们是怎么把这个仓库一步一步开发出来、以及后续应该怎么继续开发。

## 1. 第一步：先确认课题边界

最开始最重要的工作不是写代码，而是确认：

- 老师到底要求什么
- 什么是硬要求
- 什么只是组内讨论

最终确定的关键结论是：

- 必须基于 `openGauss`
- 没有证据表明必须上华为云

这一步决定了整个项目不走偏。

## 2. 第二步：先把数据库底座跑通

在写前后端之前，我们先做了数据库最小闭环：

- Docker 跑 openGauss
- 建 `hrms` 库
- 建核心表
- 插入样例数据
- 写验证脚本

原因很简单：

- 没有稳定数据库，后端接口就容易乱
- 表结构没定，前端字段也会反复改

## 3. 第三步：先把规范写清楚

在大规模写代码之前，我们先沉淀了：

- 数据库规格说明
- API 规格说明
- 模块分工说明
- 最小验证标准
- 项目路线图

这一步的意义是：

- 先把边界冻结
- 避免边开发边互相打架

## 4. 第四步：做最小后端

后端不是一下子追求“最优技术栈”，而是先追求：

- 能跑
- 能测
- 能演示
- 能和当前 openGauss 环境稳定协作

所以后端采用了：

- Python 最小 HTTP 服务
- `docker exec + gsql`

这是一条课程项目友好的路径。

## 5. 第五步：做最小前端

等后端能跑起来后，再去做前端。

前端选型时优先考虑的是：

- 能做管理后台
- 组件成熟
- 表格表单顺手
- CLI 启动方便

最终选成：

- Vue 3
- Vite
- Element Plus
- UnoCSS

## 6. 第六步：把脚本和文档统一起来

当数据库、后端、前端都能跑以后，接下来的重点就不再是“能不能跑”，而是：

- 别让仓库越来越乱
- 别让组员和 AI 反复找入口

所以做了两件事：

1. 所有文档收进 `docs/`
2. 所有脚本收进 `ops/`

这一步是为了让后续开发和维护更顺。

## 7. 现在这个仓库应该怎么继续开发

后续建议按这个顺序推进：

### A. 先把现有闭环补完整

- 员工页完善
- 请假审批补动作
- 审计页补出来
- 用户/角色页补出来

### B. 再补数据库字段升级

- 员工更真实的人事字段
- 请假审批字段
- 组织信息补全字段

### C. 再做亮点模块

- 备份恢复
- AI 查询助手
- AI 审计解释

## 8. 开发时的推荐习惯

### 每次改代码前

先看：

- `AGENT.md`
- `README.md`
- `docs/operations/STARTUP_GUIDE.md`

### 每次开发前

先确认：

- Docker 是否正常
- 数据库是否可连接
- 后端是否可访问
- 前端是否能启动

### 每次开发后

至少做：

- 数据库验证
- 后端 smoke test
- 前端页面打开检查

## 9. 这个仓库为什么适合 AI 协作

因为它已经有：

- 明确目录边界
- 启动脚本
- 规范文档
- 路线图
- CLI-first 入口

这意味着 AI 不需要靠猜，就能比较稳定地继续推进工作。

## 10. 最后一句话

这个仓库的开发策略不是：

> 先把所有高级功能想清楚再动手。

而是：

> 先建立最小可运行闭环，再不断把闭环补完整、补好看、补亮点。
