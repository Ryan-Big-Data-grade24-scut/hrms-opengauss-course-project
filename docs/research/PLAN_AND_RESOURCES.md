# 项目计划与资料索引

## 1. 基于现有背景的判断

结合你们的聊天记录、老师要求和现有 `docs/course_materials/project_inputs/Backup_Recovery_System_Architecture.md`，我建议把主线定成：

**主线基本盘：企业人力资源管理系统（openGauss 版）**

原因：

1. 更贴合课程评分点。老师看重的是数据库设计、CRUD、查询、权限、界面、手册与演示，不是工业级备份产品本身。
2. 人力资源系统业务实体天然清晰，ER 图、建表 SQL、索引、查询演示都更容易做完整。
3. 你们已经有“备份恢复”方向的架构草稿，可以作为亮点模块嵌入，而不是把整套项目风险都压在它上面。
4. 你们偏向 `C++17 + CMake`，纯做备份恢复平台会把时间大量消耗在底层工具封装、任务编排和异常恢复上，不利于短周期交付。

**建议最终方案：**

`企业人力资源管理系统 + openGauss 备份恢复管理子模块`

也就是：

- 主系统负责拿分
- 备份恢复模块负责亮点

这样既稳，又不会没有特色。

## 2. 基本盘方向

### 2.1 目标

做出一个老师一眼能看懂、能演示、能打分的数据库应用系统：

- 多角色登录
- 完整 CRUD
- 至少 3 类以上组合查询
- 清晰的权限控制
- 有数据库设计文档、建表 SQL、索引设计、测试数据
- 前后端能完整跑通

### 2.2 推荐业务边界

建议角色固定为 3 类：

- `系统管理员`：用户、角色、菜单、审计日志
- `HR 管理员`：员工、部门、岗位、招聘、调岗、离职
- `普通员工/主管`：个人信息查看、请假/审批、培训记录查看

建议先做 7 个核心实体：

- `employee`
- `department`
- `position`
- `attendance_or_leave`
- `recruitment`
- `transfer_or_promotion`
- `user_role_permission`

建议把“工资”“绩效”“培训”放在第二阶段，不要一开始铺太大。

### 2.3 技术落地建议

如果坚持 C++：

- 后端：`Drogon + C++17 + CMake`
- 数据库访问：`libpqxx`
- JSON：`nlohmann/json` 或 Drogon 自带能力
- 鉴权：`JWT + RBAC`
- 日志：`spdlog`
- 数据库：`openGauss`
- 环境：优先 `Docker / 容器化 openGauss`

前端如果只是课设，建议别追求复杂框架创新，重点是：

- 登录页
- 后台布局页
- 表格增删改查
- 查询筛选
- 角色权限差异展示

### 2.4 4 人分工建议

A. 数据库与 SQL

- 概念模型、逻辑模型、物理模型
- 建表 SQL、索引 SQL、初始化数据
- 典型查询 SQL、存储过程/触发器（如需）

B. 后端接口与权限

- 登录、JWT、RBAC
- 员工/部门/岗位/请假等接口
- 统一错误处理、分页、查询过滤

C. 前端页面与联调

- 登录
- 用户管理页
- 员工管理页
- 查询统计页

D. 测试、文档、备份恢复亮点模块

- 测试脚本
- 演示流程
- 系统手册
- 备份/恢复/审计子模块

### 2.5 两周 MVP 节奏

#### 第 1 周

- 定题与业务边界冻结
- ER 图与表结构冻结
- openGauss 环境跑通
- 登录、角色、员工、部门 4 个最小模块接口跑通
- 前端做登录页和 2 个管理页

#### 第 2 周

- 完成剩余核心 CRUD
- 完成组合查询、分页与模糊搜索
- 加权限控制
- 补测试数据
- 录第一次演示视频

#### 第 3 周

- 接入备份恢复子模块
- 加审计日志
- 优化 SQL、索引、查询响应
- 开始补手册与答辩材料

#### 第 4 周

- 收口
- 查漏补缺
- 做演示剧本
- 准备最终提交包

## 3. 灵感探索方向

这些方向适合作为加分点，但都不应该破坏基本盘进度。

### 3.1 最推荐的探索

**把“备份恢复”做成人力系统的运维子模块**

你们可以这样讲故事：

- 本项目主系统是企业人力资源管理系统
- 为了体现 openGauss 平台能力，额外设计了数据库备份恢复与审计模块
- 系统管理员或 DBA 可以对 HR 库执行备份、恢复、任务查询、日志审计

这样亮点很自然，而且和你们现有资料完全兼容。

### 3.2 可选探索

- 定时备份任务
- 备份文件校验和
- 审计日志查询
- 统计看板
- 员工异动历史时间线
- 导入导出 Excel/CSV

### 3.3 不建议当前阶段重押的方向

- 纯备份恢复管理系统作为唯一主线
- 一上来做微信小程序
- 自己造复杂前端框架
- 一开始把绩效、薪资、招聘、培训、审批、聊天全塞进去

这些都容易把你们拖进“看起来很大，实际拿分不稳定”的坑里。

## 4. 具体实践最佳实践

### 4.1 openGauss / GaussDB 侧

1. **环境优先容器化**
   - 先用容器把 openGauss 跑起来，减少环境问题。
   - 本地保存了 `opengauss-container-installation.html`。

2. **备份恢复优先用官方工具链**
   - 先落 `gs_dump / gs_restore` 的逻辑闭环，再考虑扩展 `gs_probackup`。
   - 课设演示重点是“能备份、能恢复、能记录”，不是做企业级容灾平台。

3. **数据库设计要保守**
   - 尽量使用明确的数据类型，如整数、日期、定长/变长字符。
   - 少用大字段、JSON、无约束自由结构。
   - 表设计要服务查询场景，而不是只追求抽象。

4. **索引围绕真实查询建**
   - 按照“登录、员工检索、部门筛选、时间范围查询、日志查询”这些演示场景设计索引。
   - 不要把每一列都建索引。

5. **查询一定做分页、筛选、排序**
   - 老师最容易看到的就是这些。
   - 组合查询优先：姓名/工号模糊 + 部门 + 岗位 + 入职时间范围。

6. **权限模型保持简单清晰**
   - 采用标准 `user-role-permission`。
   - 如果时间紧，只做到页面级/接口级权限即可。

### 4.2 C++ 后端侧

1. **分层明确**
   - `controller -> service -> repository -> db`
   - 不要把 SQL 直接写满控制器。

2. **先写统一返回结构**
   - `code / message / data`
   - 分页统一成 `list + total`

3. **错误处理先统一**
   - 登录失败、参数错误、权限不足、数据库异常都要有固定返回格式。

4. **查询接口优先于复杂写接口**
   - 课设演示中，查询、筛选、统计往往比复杂事务更容易体现完成度。

5. **备份恢复模块独立成 service**
   - `BackupService`
   - `RestoreService`
   - `AuditService`
   - 后面就算换成纯 HR 系统，也不会污染主体业务。

## 6. 我给你们的最终建议

如果目标是“高概率顺利完成并拿到比较稳的成绩”，建议立刻执行下面这个组合：

**方案：**

- 主项目：`企业人力资源管理系统`
- 数据库：`openGauss`
- 后端：`C++17 + Drogon + libpqxx + CMake`
- 亮点：`备份恢复与审计子模块`

**不要做的事：**

- 不要把纯备份恢复当唯一主线
- 不要把业务面铺太大
- 不要过早追求复杂前端
- 不要为了“创新”牺牲演示闭环

**最重要的判断标准只有一个：**

老师现场看时，能不能在 3 到 5 分钟内清楚看到：

- 有数据库设计
- 有权限
- 有 CRUD
- 有查询
- 有 openGauss 特征
- 有一点亮点

如果都能看到，这个项目就已经非常对路了。
